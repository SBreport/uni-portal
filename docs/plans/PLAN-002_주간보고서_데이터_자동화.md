# PLAN-002 · 주간 보고서 데이터 자동화

> 작성일: 2026-04-19
> 선행: 주간 보고서 Phase 1 (CRUD + 공유 링크 + 대시보드 v1) 완료
> 목적: 현재 100% 수동 입력인 보고서를 **일부 DB 자동 연동 + 일부 수동**의 하이브리드로 전환

---

## 1. 배경과 목표

### 현재 상태 (Phase 1 완료)
- `weekly_reports` 테이블: `data_json` 한 컬럼에 섹션 5개 전부 직렬화 저장
- 모든 입력은 사용자 수동 타이핑
- 대시보드 추이 차트는 수동 입력 누적분을 그리는 구조 (즉, 사람이 꾸준히 써야 의미 생김)
- 플레이스/웹사이트는 `place_daily` / `webpage_daily`에 이미 일별 데이터 풍부

### 목표
1. **플레이스·웹사이트 숫자 필드**: DB에서 자동 집계 → 사람은 상태 설명만 쓰면 됨
2. **수동/자동 구분 유지**: 자동 계산된 값과 사람이 덮어쓴 값(override) 분리 보관
3. **누적 데이터 활용**: 매주 자동 집계가 쌓이면 대시보드 추이 차트가 실시간으로 의미 있어짐
4. **향후 확장**: 블로그 배포·블로그 상위노출·함께찾는 섹션도 같은 패턴으로 확장

## 3. 선결 과제 (다음 세션 시작 시 결정)

### 3.1 플레이스 "휴식" 정의 🟡
현재 `pausedList` 필드는 있지만 DB 지원 없음. 옵션:

| 옵션 | 설명 | 구현 |
|---|---|---|
| A | 관리자가 수동 지정 (지점별 `is_paused` 플래그) | 지점 테이블에 컬럼 추가 |
| B | N일 연속 미노출 자동 감지 + 관리자 확정 | 룰 + 수동 확인 혼합 |
| C | 별도 `branch_status` 테이블 (active/paused/stopped) | 상태 머신 |

**권장 검토**: A + B 혼합 (자동 후보 제시 → 사람이 확정)

### 3.2 웹사이트 "이탈 / 미점유" 정의 🟡

| 용어 | 후보 정의 |
|---|---|
| 이탈 | 직전 주차에 노출됐으나 이번 주 0일 노출된 지점/키워드 |
| 미점유 | 최근 N주 동안 한 번도 노출되지 않은 지점/키워드 |
| 노출 | 이번 주 1일 이상 노출된 지점/키워드 |

**질문**: 기준일수(예: 주 1일 이상 노출 = visible / 주 0일 = dropped)를 얼마로 볼지.

### 3.3 주간 집계 기준일
- 주차 = 월요일 ~ 일요일 (이미 확정)
- "금일 집계" vs "주간 평균" — 어느 값을 KPI로 삼을지
- **금일 상위노출 지점 수** 라벨 기준이라면 **마지막 측정일 스냅샷**이 맞을 듯

### 3.4 집계 트리거
- A. **수동**: 작성 탭 상단 "자동 채우기" 버튼 → 클릭 시 해당 주차 갱신
- B. **자동**: 매주 화요일(보고 날) 자동 집계 스크립트
- C. **보고서 생성 시**: 신규 주차 보고서 생성 POST 호출 시 즉시 자동값 삽입

**권장**: **A + C** — 생성 시 자동 + 이후엔 수동 새로고침

## 4. 데이터 모델 확장안

### 옵션 A: `data_json` 내부에 auto/override 분리 (단순)
```json
{
  "place": {
    "total": "41",             // 최종 표시값 (override 있으면 override, 없으면 auto)
    "total_auto": "41",        // 자동 계산값 (매번 덮어씀)
    "total_override": null,    // 사람이 수정한 값 (null이면 auto 사용)
    "summary": "...",          // 수동 텍스트는 그대로
    "auto_updated_at": "2026-04-19T10:00:00"
  }
}
```

**장점**: 기존 스키마 무변경. 프론트에서 렌더 시 `override ?? auto`.
**단점**: 필드 수 증가. JSON 파싱 약간 복잡.

### 옵션 B: 별도 테이블 `weekly_report_autofill` (복잡)
```sql
CREATE TABLE weekly_report_autofill (
    week_start DATE,
    section TEXT,
    field TEXT,
    auto_value TEXT,
    computed_at TIMESTAMP,
    PRIMARY KEY (week_start, section, field)
);
```

**장점**: 관리·조회 명확, auto 값 히스토리 가능
**단점**: 복잡도 ↑, API 2번 호출

### 권장: **옵션 A** — 단순성 우선. 히스토리 추적은 나중에 필요해지면 B 이행.

## 5. UI 변경 설계

### 5.1 작성 탭 각 필드
```
[총 지점 수]
┌────────┐
│  41    │ 🤖 자동 [수정]
└────────┘
       auto: 41  ·  4/19 10:00 갱신
```
- 필드에 자동값 채워짐 + 작은 배지 "자동"
- "수정" 클릭 → input 활성화 → 저장 시 override에 들어감
- override 있으면 배지가 "수정됨 [되돌리기]"

### 5.2 상단바에 "자동 채우기" 버튼 (editor+)
- 클릭 시 해당 주차 auto 필드 전부 재계산
- override 값은 유지 (덮어쓰지 않음)
- 결과: "5개 필드 갱신됨" 토스트

### 5.3 결과 탭 / 공개 페이지
- 최종 표시값만 보여줌 (auto인지 override인지 내부 구분만)
- 사용자가 볼 필요 없음
