# PLAN-004 · 탐색기 기능 개선

> 작성일: 2026-04-21
> 선행: PLAN-003 Phase 2 완료 (place_daily·webpage_daily FK 100% 채움, branch_resolver 공용화)
> 목적: ExplorerView의 사용성·기능성을 개선한다. 구체적인 개선 항목은 다음 세션 시작 시 사용자와 확정한다.

---

## 1. 개요

탐색기(ExplorerView)는 포탈에서 "지점·시술·장비·논문"을 4개 축으로 교차 탐색하는 허브 화면이다. 현재 v1 형태로 작동하고 있으며, 이번 세션(2026-04-21)에서 다음 두 가지 소규모 개선이 완료됐다.

- 지점명 옆 지역 태그 제거
- 웹페이지 요약 카드 이분법 표시(`상위노출 중` / `미노출 중`)

Phase 2 FK 구조화 이후 탐색기가 더 정확하고 빠른 데이터를 다룰 수 있는 기반이 마련됐다. PLAN-004는 이 기반 위에서 탐색기를 사용자 입장에서 실질적으로 더 유용하게 만드는 개선을 계획한다.

**중요**: 이 문서는 **논의 프레임워크**다. 아래 개선 후보와 항목들은 브레인스토밍 목록이며, 실제 작업 범위는 다음 세션 시작 시 사용자와 합의 후 확정한다.

## 2. 현재 탐색기 구조

### 2.1 ExplorerView.vue 4개 탭

| 탭 | 설명 | 데이터 소스 |
|---|---|---|
| 지점별 | 지점을 선택하면 해당 지점의 모든 연관 데이터를 한 화면에 집약 | `place_daily`, `webpage_daily`, `blog_articles`, 이벤트, 장비 등 |
| 시술별 | 시술명으로 관련 장비·지점·블로그를 탐색 | `encyclopedia` (시술백과) |
| 장비별 | 장비명으로 관련 시술·지점을 탐색 | `equipment` DB |
| 논문 | 논문 키워드·제목으로 검색 | `papers` DB |

### 2.2 api/routers/explorer.py 주요 엔드포인트

| 엔드포인트 | 역할 |
|---|---|
| `GET /explorer/branches` | 지점 목록 (short_name 포함) |
| `GET /explorer/by-branch` | 지점별 상세 — 연관 데이터 집약 반환 |
| `GET /explorer/by-procedure` | 시술별 탐색 |
| `GET /explorer/by-equipment` | 장비별 탐색 |
| `GET /explorer/papers` | 논문 검색 |

### 2.3 지점별 상세 화면(`by-branch`) 표시 데이터

지점을 선택하면 아래 7개 영역 데이터를 한 화면에 표시한다.

| 영역 | 내용 |
|---|---|
| 장비 | 해당 지점 보유 장비 목록 |
| 이벤트 | 현재·과거 이벤트 목록 |
| 블로그 | 최근 발행 블로그 포스트 |
| 플레이스 | 플레이스 순위 현황 (ref_cutoff = 오늘-2일 기준) |
| 웹페이지 | 웹사이트 상위노출 현황 |
| 카페 | 카페 게시물 |
| 민원 | 민원 현황 |

### 2.4 이번 세션에서 정비된 기반

탐색기 개선 시 활용 가능한 기반 자산:

| 자산 | 내용 |
|---|---|
| `shared/branch_resolver.py` | 지점명→ID 해석 공용 유틸. 탐색기 쿼리에서 재사용 가능 |
| `place_daily.evt_branch_id` | 100% FK 채움 완료. JOIN 기반 빠른 집계 가능 |
| `webpage_daily.evt_branch_id` | 100% FK 채움 완료. 동일 |
| `evt_branches.aliases` | 별칭 매칭 완료. 검색 UX에서 별칭으로도 지점 찾기 가능 |
| `evt_branches.is_paused` | 휴식 상태 플래그. 탐색기 필터로 활용 가능 |

## 5. 작업 시 주의사항

다음 세션에서 탐색기를 개선할 때 반드시 유지해야 할 사항들이다.

### 5.1 FK 구조 활용

이번 세션에서 `place_daily`·`webpage_daily` 모두 `evt_branch_id` FK 100%를 달성했다. 탐색기 신규 쿼리는 `branch_name LIKE` 대신 `evt_branch_id = ?` JOIN 방식을 우선 사용할 것.

```sql
-- 금지: 문자열 매칭
WHERE branch_name LIKE '%광주%'

-- 권장: FK JOIN
WHERE place_daily.evt_branch_id = :branch_id
```

### 5.2 광주/경기광주 분리 유지

`shared/branch_resolver.py`가 `LENGTH(short_name) DESC` 우선 매칭으로 두 지점을 구분한다. 탐색기 코드에서 직접 LIKE 매칭을 새로 추가하지 말 것. 반드시 resolver를 경유할 것.

### 5.3 휴식 상태 표시 유지

`evt_branches.is_paused`가 이번 세션에 도입됐다. 탐색기 지점 목록에서 휴식 중인 지점을 별도 표시하거나 필터 가능하도록 유지할 것. 휴식 지점을 목록에서 무조건 숨기지 말 것.

### 5.4 ref_cutoff 기준일 준수

`by-branch` 엔드포인트는 이번 세션에서 `ref_cutoff = today - 2days`로 확정됐다. 신규 지점별 조회를 추가할 때 동일한 cutoff를 적용할 것. 기준일 변경은 사용자 확인 후 진행할 것.

### 5.5 디자인 원칙

- 다열 그리드 지양 — 탐색기 지점 카드는 현재 1열 중심 구조를 유지한다.
- 여백 과도 금지 — 정보 밀도 우선, `p-3`/`space-y-3` 기본.
- 로딩·빈값·에러 상태 4종 처리 필수.
